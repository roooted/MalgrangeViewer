# Техническое задание для Codex

## Название проекта
Malgrange Graph Visualizer

## Цель
Разработать локальное веб-приложение для создания и редактирования ориентированного графа, синхронизированного с матрицей смежности, и поиска компонент сильной связности с помощью стандартного алгоритма Мальгранжа.

Приложение должно предоставлять визуальный редактор графа, редактируемую матрицу смежности, интерактивное создание и удаление дуг, отмену и повтор действий создания и удаления связей, а также визуальную раскраску найденных компонент сильной связности.

---

## Технологический стек
- React
- Vite
- TypeScript
- React Flow

Приложение должно работать локально в браузере.
Бэкенд не требуется.
База данных не требуется.

## Исходный визуальный ориентир
Визуальные состояния редактора и демонстрации результатов алгоритма были разделены на два макета. Цвета в итоговом состоянии используются как функциональный признак принадлежности вершин и дуг к одной компоненте. Макеты используются как точный визуальный ориентир.

Важные замечания по макетам:
- они задают общую компоновку, тёмный стиль, основные зоны интерфейса и состояния взаимодействия;
- они задают все тонкости подсветки рёбер, дуг, ячеек матрицы и списка компонентов
- они **не** задают точное выравнивание компонентов интерфейса;

Макеты нужно использовать как точное визуальное направление, отходить от них только при необходимости.

---

## Функциональное назначение
Приложение предназначено для того, чтобы:
- позволять пользователю задавать количество вершин;
- автоматически размещать вершины по окружности при запуске;
- позволять создавать и удалять ориентированные дуги как на графе, так и через матрицу смежности;
- поддерживать петли;
- запускать стандартный алгоритм Мальгранжа на текущем ориентированном графе;
- показывать список компонент сильной связности;
- раскрашивать вершины, дуги, ячейки матрицы и поле компонента в свой контрастный цвет.

## Базовая модель графа
### Тип графа
Граф является **ориентированным**.

### Вершины
- При запуске приложение должно создавать заранее определённое количество изолированных вершин.
- Вершины должны автоматически размещаться по окружности.
- Подписи вершин должны иметь формат: `x1`, `x2`, `x3`, ...
- Минимально уместное количество вершин — 2.
- Максимальное уместное количество вершин — 20.
- Количество вершин по умолчанию может быть выбрано в реализации, но должно быть явно вынесено в константу.

### Изменение количества вершин
- В интерфейсе должен быть отдельный числовой input для количества вершин.
- Изменение значения в input само по себе не должно сразу пересоздавать граф.
- После изменения значения в input должно появляться окно подтверждения с заголовком "Reconstruct current graph?" с английскими опциями:
  - `Confirm`
  - `Cancel`
- Если пользователь выбирает `Confirm`, граф должен быть полностью пересоздан.
- Если пользователь выбирает `Cancel`, пересоздание не выполняется, а текущее состояние графа сохраняется.
- При подтверждённом пересоздании:
  - все вершины создаются заново;
  - все дуги удаляются;
  - матрица смежности очищается и создаётся заново;
  - история undo/redo очищается.

## Редактирование графа

### Создание и удаление дуг через матрицу смежности
Матрица смежности должна быть редактируемой.

Правила:
- Клик по ячейке матрицы переключает её значение.
- Если текущее значение ячейки равно `0`, оно становится `1`, и соответствующая ориентированная дуга создаётся.
- Если текущее значение ячейки равно `1`, оно становится `0`, и соответствующая ориентированная дуга удаляется.
- Изменения должны работать с двусторонней синхронизацией:
  - matrix -> graph;
  - graph -> matrix.

Интерпретация:
- Ячейка `(i, j)` соответствует дуге `xi -> xj`.
- Диагональная ячейка `(i, i)` соответствует петле на вершине `xi`.

### Создание дуг на графе
Пользователь также должен иметь возможность создавать дуги напрямую на графе.

Правила:
- Клик по вершине запускает режим создания дуги из этой вершины.
- Сразу должна появляться временная пунктирная стрелка.
- Пока режим создания дуги активен, эта временная пунктирная стрелка должна в реальном времени следовать за курсором мыши.
- Создание дуги завершается, когда:
  - пользователь кликает по целевой вершине, или
  - пользователь нажимает `Esc`.
- Если пользователь кликает по другой вершине, создаётся ориентированная дуга из исходной вершины в выбранную целевую.
- Если пользователь повторно кликает по той же вершине, создаётся петля.
- После успешного создания временная пунктирная стрелка становится обычной сплошной дугой.
- После отмены через `Esc` временная пунктирная стрелка исчезает.
- Повторное создание точно такой же ориентированной дуги запрещено.

Правила уникальности:
- `xi -> xj` и `xj -> xi` — это разные дуги.
- Запрещены только точные дубликаты.
- Петля тоже уникальна для каждой вершины.

### Удаление дуг на графе
Дуги должны удаляться не только через матрицу, но и напрямую на графе.

Правила:
- Клик по существующей дуге выделяет её.
- Выделенная дуга становится пунктирной.
- Нажатие `Delete` удаляет выделенную дугу.
- При удалении дуги на графе соответствующая ячейка матрицы должна сразу переключаться в `0`.

### Undo и redo
В интерфейсе должны быть кнопки:
- `<` для undo;
- `>` для redo.

Undo/redo должны поддерживать как минимум такие действия редактирования:
- создание дуги;
- удаление дуги;
- создание петли;
- удаление петли;
- переключение дуги через матрицу.

Примечания:
- Undo/redo обязательны для действий создания и удаления связей.
- Очистка и пересоздание графа из-за изменения числа вершин должны сбрасывать историю.
- Undo/redo для результата работы алгоритма не требуется.

## Состояния наведения, активности и выбора
Состояния взаимодействия, показанные в wireframe, должны быть реализованы.

### Состояния вершины
- При наведении курсора вершина подсвечивается, пока курсор находится над ней.
- После клика по вершине как по источнику дуги подсвечивается ярче.
- Усиленная подсветка остаётся активной до тех пор, пока:
  - дуга не будет создана, или
  - создание дуги не будет отменено через `Esc`.

### Состояние временной дуги
- Во время создания связи дуга, следующая за курсором, должна быть пунктирной.
- Она остаётся пунктирной до завершения создания или отмены.
- После создания дуга становится сплошной.

### Состояния дуги
- При наведении курсора дуга подсвечивается, пока курсор находится над ней.
- При наведении на дугу соответствующая ячейка матрицы должна одновременно подсвечиваться.
- При клике дуга становится выбранной и пунктирной.
- Когда дуга выбрана, соответствующая ячейка матрицы тоже должна визуально переходить в выбранное состояние с пунктирным стилем, как в идее wireframe.

### Состояния ячейки матрицы
Матрица должна визуально отражать взаимодействия с графом.

Обязательная синхронизация:
- hover on edge -> highlight corresponding matrix cell;
- select edge -> mark corresponding matrix cell as selected;
- create/delete edge from graph -> immediately update the matrix value;
- toggle matrix cell -> immediately create/delete the corresponding graph edge.

## Специальные правила отрисовки графа
### Обычные ориентированные дуги
- Ориентированные дуги должны отображаться со стрелками.
- Визуальный стиль должен соответствовать тёмному академическому интерфейсу из макетов.

### Взаимные дуги между двумя вершинами
Если между одной и той же парой вершин существуют две противоположно направленные дуги, они должны отображаться как две изогнутые стрелки, образующие визуально разделённую пару, похожую на эллипс.

Конкретно:
- `xi -> xj` и `xj -> xi` не должны накладываться в одну прямую линию;
- обе дуги должны оставаться различимыми;
- степень изгиба должна быть достаточной для удобного чтения.

### Петли
- Петли допустимы.
- Петля создаётся выбором одной и той же вершины как источника и как цели.
- Петля должна отображаться как небольшая дуга рядом с вершиной.
- Направление и положение петли должны оставаться визуально согласованными.
- По возможности петлю лучше размещать с внешней стороны относительно центра графа.

---

## Матрица смежности
Интерфейс должен показывать матрицу смежности с самого начала, даже когда граф пуст по дугам.

Требования:
- размер матрицы — `N x N`, где `N` — текущее количество вершин;
- строки и столбцы подписаны именами вершин;
- значения бинарные: `0` или `1`;
- все ячейки редактируются одним кликом;
- диагональные ячейки управляют петлями;
- матрица всегда должна точно отражать текущее состояние графа.

Матрица является частью основного редактора, а не debug-панелью.

## Требования к алгоритму
### Какой алгоритм использовать
Нужно использовать **стандартный алгоритм Мальгранжа** для поиска компонент сильной связности.

Нельзя заменять его другим алгоритмом.
Нельзя получать тот же результат через другой SCC-алгоритм во внутренней реализации.

### Входные данные алгоритма
Алгоритм должен запускаться на графе, который в данный момент находится в редакторе.

### Вывод результата алгоритма
После нажатия на кнопку `Find Components`:
- результат должен показываться сразу;
- промежуточные вычисления показывать **не нужно**;
- требуется только итоговый список компонент сильной связности.

### Формат результата
После расчётов в списке компонент каждая компонента должна подсвечиваться своим цветом и отображаться как на макетах.


### Визуальный результат 
Для каждой найденной компоненты сильной связности:
- все вершины этой компоненты должны окрашиваться в один отдельный контрастный цвет;
- все дуги, концы которых принадлежат этой компоненте, должны окрашиваться в тот же цвет компоненты;
- каждая компонента должна иметь собственный уникальный цвет.
- Поле компоненты (на макете в правом нижним углу) также окрашивается.

Никаких дополнительных представлений результата не требуется.
Никаких промежуточных таблиц не требуется.
Граф конденсации не требуется.
Пошаговое объяснение алгоритма не требуется.

## Поведение для несвязных графов
Приложение должно поддерживать запуск алгоритма на несвязных ориентированных графах.

Это допустимо и ожидаемо.

Примечания для реализации:
- изолированные вершины допустимы;
- вершины без исходящих дуг допустимы;
- вершины без входящих дуг допустимы;
- графы с несколькими несвязанными группами допустимы;

Алгоритм всё равно должен возвращать корректные компоненты сильной связности для всех таких случаев.

## Рекомендуемая архитектура
Нельзя концентрировать всю логику в одном файле.

Кодовая база должна быть разделена на небольшие и понятные файлы.

### Обязательное разделение ответственности
Проект нужно разделить как минимум на такие логические слои:

#### 1. UI layer
Отвечает за:
- отображение элементов управления;
- отображение графа;
- отображение матрицы;
- отображение результатов;
- обработку пользовательских взаимодействий.

#### 2. Graph state layer
Отвечает за:
- хранение вершин и дуг;
- состояние выбора;
- состояние hover;
- состояние временного создания дуги;
- состояние undo/redo;
- синхронизацию между графом и матрицей.

#### 3. Algorithm layer
Отвечает за:
- преобразования графа;
- создание списка смежности;
- создание обратного графа;
- реализацию алгоритма Мальгранжа;
- возврат результата в формате данных, не зависящем от UI.

Слой алгоритма не должен зависеть от объектов React Flow.
UI-слой не должен содержать в себе сам алгоритм.

## Детали взаимодействия и граничные случаи
### Обязательные граничные случаи
Реализация должна корректно обрабатывать:
- пустое множество дуг;
- граф только с односторонними дугами;
- граф с противоположно направленными дугами между двумя вершинами;
- петли;
- несвязный граф;
- изолированные вершины;
- повторные попытки создать одну и ту же дугу;
- отмену создания дуги через `Esc`;
- удаление выбранной дуги клавишей `Delete`;
- переключение ячейки матрицы для существующей и несуществующей дуги.

### Поведение выбора
- Одновременно для удаления должна быть выбрана не более чем одна дуга
- Начало нового создания дуги должно очищать предыдущее временное состояние создания.
- Нажатие `Esc` должно отменять только текущий режим создания дуги.

### Пересчёт результата
Результат поиска компонент сильной связности не должен автоматически обновляться после каждого изменения графа.
Нужно пересчитывать его только после нажатия `Find Components`.

Однако если после показа результата граф изменился 
- необходимо очистить предыдущую подсветку и список компонент;
- результат может быть явно пересчитан только при следующем нажатии кнопки.

Таким образом: после любого редактирования графа предыдущая визуализация SCC должна очищаться, пока пользователь снова не нажмёт `Find Components`.

## Критерии приёмки
Задача считается выполненной, если соблюдены все условия ниже.

### Редактирование
- Пользователь может задавать количество вершин.
- Изменение количества вершин происходит только через подтверждение `Confirm` / `Cancel`.
- После подтверждённого изменения количества вершин граф пересоздаётся, матрица очищается.
- Вершины размещаются по окружности.
- Пользователь может создавать дуги через матрицу.
- Пользователь может удалять дуги через матрицу.
- Пользователь может создавать дуги через взаимодействие с графом.
- Пользователь может создавать петли, выбирая одну и ту же вершину как источник и цель.
- Повторные ориентированные дуги не создаются.
- Пользователь может выбрать дугу и удалить её клавишей `Delete`.
- Undo и redo работают для действий создания и удаления дуг.

### Синхронизация
- Граф и матрица остаются синхронизированными в обе стороны.
- Hover и выбор дуги отражаются в матрице.
- Переключение ячейки матрицы сразу отражается на графе.

### Визуальные состояния
- Работает hover по вершине.
- Работает hover по дуге.
- Выбранная исходная вершина становится ярче во время создания дуги.
- Временная дуга пунктирная во время создания.
- Выбранная дуга пунктирная.

### Алгоритм
- Кнопка `Find Components` запускает стандартный алгоритм Мальгранжа.
- Каждая компонента окрашивается в свой контрастный цвет.
- Вершины и внутренние дуги каждой компоненты окрашиваются соответствующим цветом.
- Приложение корректно работает для несвязных графов.

### Качество кода
- Код разделён на небольшие и понятные файлы.
- Логика алгоритма отделена от UI.
- Русские комментарии добавлены в нужных местах.
- TypeScript-типы используются явно и понятно.
